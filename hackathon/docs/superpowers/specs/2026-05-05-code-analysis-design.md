# Code Analysis (Smells + Security) — Design

**Date:** 2026-05-05
**Status:** Approved (design phase)
**Owner:** vasu.kothapalli@moodys.com
**Builds on:** `2026-05-05-code-scanner-design.md` (already shipped)

## Goal

Add an opt-in analysis pass to the existing Code Scanner CLI that produces:

1. **Code smell findings** for C# source files: long functions, deep nesting, long parameter lists. Roslyn-backed (exact AST).
2. **Security findings** across all text files: hardcoded secrets via regex, dangerous-function calls per language.

Both feed structured arrays into the existing JSON output without breaking its shape.

## Non-Goals

- Smell or security analysis for languages other than C# (smells) or the listed extensions (security). Architecture leaves a hook for more, but v1 ships only what's listed.
- Taint analysis, dataflow, or any AST-based security checks. Security v1 is pattern-based only.
- Auto-fixing or quick-fix suggestions. Findings are read-only.
- Configurable thresholds via CLI. v1 uses fixed values from the design tables below.
- A separate `analyze` subcommand. Analysis is a phase of the existing `scan` invocation.

## CLI Surface (additions)

| Flag | Default | Purpose |
|---|---|---|
| `--smells` | off | Run Roslyn-based smell analyzer on every `.cs` file |
| `--security` | off | Run regex-based secret + dangerous-function scanners on all text files |
| `--analyze` | off | Shorthand for `--smells --security` |
| `--security-skip <glob>` | (none, additive) | Additional path globs to skip in the security scan only |

All other CLI behavior unchanged.

**Performance:** when both flags are off, the binary skips loading Roslyn and security pattern lists entirely (`--smells`/`--security` are the trigger).

## Code Smell Detection (C# only)

### Thresholds and severity buckets

| Smell | Threshold | low | medium | high |
|---|---|---|---|---|
| Long function | > 50 lines | 51–75 | 76–150 | 151+ |
| Deep nesting | > 4 levels | 5–6 | 7–8 | 9+ |
| Long parameter list | > 5 params | 6–7 | 8–10 | 11+ |

### What counts as a "function"

- Methods (`MethodDeclarationSyntax`)
- Constructors (`ConstructorDeclarationSyntax`)
- Local functions (`LocalFunctionStatementSyntax`)
- Finalizers (`DestructorDeclarationSyntax`)
- Operators (`OperatorDeclarationSyntax`, `ConversionOperatorDeclarationSyntax`)
- **Skipped:** lambdas (`ParenthesizedLambdaExpressionSyntax`, `SimpleLambdaExpressionSyntax`) and anonymous methods. They are typically one-liners; treating them as functions creates noise.

### What counts as "lines"

`endLine - startLine + 1`, where the span is the entire declaration (signature through closing `}`). Expression-bodied members (`int Foo() => 42;`) count as 1 line and never trigger the threshold.

### What counts as a "nesting level"

Each of the following increments depth by 1:
- `BlockSyntax` directly inside a control-flow statement (if/else, for, foreach, while, do, switch, try, catch, finally, lock, using, fixed, unsafe)
- Local function declarations
- Lambda bodies (yes — even though we don't analyze them as functions, their body still adds nesting depth in the enclosing function)

The function's outermost `{...}` body counts as depth 1 (so threshold > 4 means a `{` 5 levels deep inside the body).

### Generated code

Files starting with `// <auto-generated>` (case-insensitive, within the first 5 lines) are skipped from smell analysis. Other generated-code markers are out of scope for v1.

## Security Scanning (regex-based, language-aware)

### Hardcoded secret patterns (v1 set)

| Subtype | Regex | Severity |
|---|---|---|
| `aws_access_key` | `\b(AKIA\|ASIA)[0-9A-Z]{16}\b` | high |
| `github_pat` | `\b(ghp\|gho\|ghu\|ghs\|ghr)_[A-Za-z0-9]{36,}\b` | high |
| `slack_token` | `\bxox[baprs]-[A-Za-z0-9-]{10,48}\b` | high |
| `private_key` | `-----BEGIN [A-Z ]*PRIVATE KEY-----` | high |
| `jwt` | `\beyJ[A-Za-z0-9_-]+\.[A-Za-z0-9_-]+\.[A-Za-z0-9_-]+\b` | medium |
| `generic_assign` | `(?i)(password\|secret\|api[_-]?key\|token)\s*[:=]\s*["'][^"'\s]{8,}["']` | medium |
| `connstr_password` | `(?i)(Password\|Pwd)=([^;'"\s]+)` | medium |

Applied to every non-binary file under the scanned root, regardless of extension. (`.gitignore`, `.env`, etc. are real risk files.)

### Dangerous-function patterns (v1 set, dispatched by extension)

| Extensions | Regex | Severity |
|---|---|---|
| `.js, .ts, .jsx, .tsx, .mjs, .cjs` | `\beval\s*\(` | high |
| `.js, .ts, .jsx, .tsx, .mjs, .cjs` | `\bnew\s+Function\s*\(` | high |
| `.js, .ts, .jsx, .tsx, .mjs, .cjs` | `\bsetTimeout\s*\(\s*['"]` | medium |
| `.js, .ts, .jsx, .tsx, .mjs, .cjs` | `\bsetInterval\s*\(\s*['"]` | medium |
| `.py, .pyi` | `\beval\s*\(` | high |
| `.py, .pyi` | `\bexec\s*\(` | high |
| `.py, .pyi` | `\bsubprocess\.[A-Za-z_]+\([^)]*shell\s*=\s*True` | medium |
| `.ps1, .psm1` | `\bInvoke-Expression\b` | high |
| `.ps1, .psm1` | `\biex\s+` | high |
| `.sh, .bash` | `^\s*eval\s+` (multiline) | high |
| `.cs` | `\bAssembly\.Load\s*\(` | low |

### Inline ignore directive

If the matched line ends (after stripping whitespace) with `codescan:ignore` (preceded by `//`, `#`, `--`, or `;` comment markers), the finding is suppressed for that line. Token is case-sensitive.

```js
const apiKey = "AKIA1234567890ABCDEF"; // codescan:ignore
```

### Snippet redaction

For every secret finding, the `snippet` field shows:
- The full source line, but the **matched secret value** is replaced with the first 4 characters + `••••REDACTED`.
- For `private_key` and connection strings: the entire match is replaced with `••••REDACTED` (no prefix preview).

The full secret is never written to JSON, stderr, or logs.

### Security scan skip rules

- Binary files (already detected by existing binary-sniff) are skipped.
- Files larger than **1 MB** are skipped. Logged in `scanned.errors` with reason `"file too large for security scan"`.
- Globs passed via `--security-skip` are skipped (matches against path relative to scan root, using `Microsoft.Extensions.FileSystemGlobbing`).

## Module Layout

New files under `src/CodeScanner/Analyzers/`:

```
src/CodeScanner/
├── Analyzers/
│   ├── ISmellAnalyzer.cs           # interface for AST-based smell detection
│   ├── ISecurityScanner.cs         # interface for text-based security detection
│   ├── CSharpSmellAnalyzer.cs      # Roslyn implementation
│   ├── SmellWalker.cs              # CSharpSyntaxWalker subclass
│   ├── SecretScanner.cs            # secret regex sweep
│   ├── DangerousFunctionScanner.cs # extension-dispatched regex sweep
│   ├── Patterns.cs                 # all regex patterns + severity constants
│   ├── IgnoreDirective.cs          # codescan:ignore parsing
│   └── AnalyzerHost.cs             # orchestrates passes, aggregates findings
└── Models.cs                       # add SmellFinding, SecurityFinding records
```

**Roslyn dependency:** `Microsoft.CodeAnalysis.CSharp` (NuGet). Pulled in unconditionally — the CLI binary loses its zero-deps property — but only used when `--smells` is set.

**Module boundaries:**
- Analyzers know nothing about the CLI or JSON. They consume `string filePath` + `string content` and return `IReadOnlyList<SmellFinding>` / `IReadOnlyList<SecurityFinding>`.
- `AnalyzerHost` orchestrates: takes a `ScanResult` plus options, runs the relevant scanners file-by-file, returns enriched findings.
- `Report.Serialize` is updated to include `smells[]` and `securityIssues[]` when present, and per-language summary blocks.
- `Cli.Execute` reads the new flags and passes them through.

## Edge Cases

| Case | Behavior |
|---|---|
| `--smells` but no `.cs` files in scan | Empty `smells[]` array |
| `--security` but only binary files | Empty `securityIssues[]` array |
| Roslyn fails to parse a `.cs` file | Logged in `scanned.errors` with `ParseException`; smells skip the file; scan continues |
| Regex match overlaps `codescan:ignore` | Suppressed; no finding emitted |
| Regex match inside a comment (no ignore directive) | Reported (known v1 limitation; future work could strip comments per language) |
| File > 1 MB, security flag on | Skipped; logged as `"file too large for security scan"` |
| File > 1 MB, smells flag on | Roslyn handles large files fine — analyzed normally |
| Same secret matches multiple patterns on same line | Deduped: highest-severity match wins; only one finding per (file, line, column) |
| `--security-skip "tests/**"` | Path globs evaluated against root-relative paths |
| Both flags off | Binary loads no analyzer code paths — JSON shape and runtime equivalent to v1 |

## Testing Strategy

### Smell analyzer (Roslyn)
- Inline source-string tests: pass a `string` of C# code into `CSharpSmellAnalyzer.Analyze(content, path)`, assert the findings list. No fixture files needed for parser-driven tests.
- Tests for each smell type at each severity bucket boundary (51, 75, 76, 150, 151 lines, etc.).
- Test that lambdas don't trigger long-function detection.
- Test that auto-generated files are skipped.

### Security scanner (regex)
- Inline content-string tests for each pattern: positive match, near-miss negative, ignore-directive suppression, snippet-redaction format.
- One fixture file per scanner under `tests/CodeScanner.Tests/fixtures/Security/` for end-to-end verification (with intentionally fake AKIA-style strings — clearly marked as test fixtures).
- Test deduplication: a line that matches `generic_assign` AND `aws_access_key` produces one finding (the higher-severity AWS one).

### Integration / CLI
- End-to-end test: build a fixture tree with smelly C# + a fake AWS key; run `--analyze`; assert both arrays in the JSON.
- End-to-end test: same tree, no flags → JSON has no `smells` or `securityIssues` keys (backwards compat).

## Open Questions

None at design time — all major choices pinned above.

## Out-of-Scope Future Work

- Configurable thresholds (`--max-function-lines`, etc.)
- Smell support for Python, JS/TS via additional analyzers
- AST-based security analysis (taint, dataflow)
- More secret patterns (Stripe keys, GCP keys, Azure SAS, Generic API keys with entropy heuristics)
- More dangerous-function patterns (Ruby `eval`, PHP `eval`, etc.)
- SARIF output format for tool interop
- Quick-fix or auto-fix suggestions
- Per-language comment stripping before regex (would reduce false positives)
